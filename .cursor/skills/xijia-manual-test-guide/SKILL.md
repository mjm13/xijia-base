# 人工验收说明生成器

## 目标

在需求收尾时，把“代码已完成”转换为“人可复现验收”的说明，直接服务 `00-workflow.mdc` 要求的 `# 验收记录` 输出。

本技能只定义验收说明结构与写法，不绑定任何技术栈。

## 何时使用

- 需求/Bug 完成后准备收尾
- 用户要求“给一份怎么验收/怎么手测”的说明
- 需要将自动化结果转化为人工可执行步骤

以下场景不使用本技能：

- 仅文档排版或注释微调且无行为变化
- 仅讨论技术方案，尚未进入收尾验收

## 输入清单（先收集再写）

1. 变更范围：模块、入口、能力点
2. 可观察行为：用户操作后能看到什么结果
3. 自动化覆盖：已执行命令、覆盖到哪些场景
4. 本次范围外：明确未做项与后续切片项

若任一输入缺失，先标记 `[待确认]`，不要臆测补齐。

## 产出位置

- 优先写入对应需求文件的 `# 验收记录`
- 若该需求仅有计划文档，则写入对应 `docs/plans/*.md`
- 多次迭代为追加，不覆盖历史记录

## 标准模板

```markdown
## 人工验收说明（Acceptance Note）

- 涉及菜单 / 模块：<moduleKey + 菜单路径；无界面写“后端 API / 无界面”>
- 改了什么功能：<一句话能力描述>
- 验收场景（人可复现）：
 1. GIVEN <前置> WHEN <操作> THEN <可观察结果>
 2. ...
- 手动验证步骤：<点击路径/输入/预期结果；无界面给接口调用示例与预期响应>
- 自动化覆盖：<测试名或命令>；未覆盖项标注“需人工点验”
- 本次范围外 / Deferred：<明确列出未做链路>
```

## 编写规则

1. 一律用“可观察结果”表达，不写实现细节。
2. 场景优先级：先主流程，再关键分支，再异常路径。
3. 自动化与人工边界清晰：自动化覆盖不代表人工可省略。
4. Deferred 必须与本次 proposal/design 的范围声明一致。

## 快速自检

- [ ] 写明了菜单/模块或“无界面”
- [ ] 至少 1 条完整 GIVEN/WHEN/THEN
- [ ] 手动步骤可由非开发同学执行
- [ ] 自动化已覆盖与未覆盖边界明确
- [ ] Deferred 没有被误写成“已完成”
