# 目标

以 **红-绿-重构** 循环驱动前端实现：先写失败验证（RED），再写最小实现（GREEN），最后清理（REFACTOR）。**禁止先写实现再补测试。**

与 `openspec-superpowers-apply` 联动：每条 In-Scope AC 必须有对应测试或明确可运行验证命令，否则不得勾选任务。

# 栈无关测试门禁

## 1) 判定测试类型

- 组件渲染、交互、状态同步：优先组件/集成级验证。
- 纯函数与数据转换：单元测试优先。
- 外部接口与浏览器环境依赖：按边界隔离，可 mock 网络与外部服务。

## 2) 强制规则

- 前端行为必须与已确认契约一致（字段语义、错误语义、权限可见性）。
- 不允许以硬编码前端逻辑替代后端业务真值。
- 出现契约冲突时，先升级规格/契约再改代码。

## 3) 执行步骤（RED -> GREEN -> REFACTOR）

1. RED：先写失败测试（交互成功、失败路径、边界场景）。
2. GREEN：补最小实现使测试通过。
3. REFACTOR：收敛重复逻辑，保持可读与可维护。

# 最小验收清单

- [ ] 每条 In-Scope AC 至少有 1 个测试或明确验证命令
- [ ] 关键交互、异常路径、边界场景均覆盖
- [ ] API 契约对齐有可执行验证
- [ ] 回归验证可重复执行

# 与技术栈技能的关系

- 本技能只定义测试原则与门禁，不绑定具体框架/命令。
- 具体测试命令、测试运行器、目录规范由已安装技术栈技能提供（例如 Vue、React、Svelte 等）。

# 禁止事项

- 先实现后测试
- 契约冲突时在前端私自“兜底修复”
- 仅以人工点点看作为唯一验证方式
