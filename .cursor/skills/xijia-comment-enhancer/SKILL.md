# AI 注释增强与维护（端点/内部分层 · 语义优先）

## 目标

为**核心业务代码**按统一维度补全或更新**语义注释**：区分端点层与内部层，`[业务逻辑]` 写业务含义而非复述代码，严禁臆测、无证据则省略。**本技能只约定注释格式与方法论，与具体语言、框架、数据库无关。**

## 何时启用（触发）

- 新增或修改**核心业务代码**（对外入口、业务处理单元、工具 / Helper、复杂私有方法）
- 用户要求**补注释、增强注释、写注释文档**
- **逻辑变更后**需追加 `[修改记录]`
- **提交前**检查目标方法注释是否完整

> 自解释的一行函数、纯数据结构、getter/setter 等**非核心**代码不强制注释。

## 方法分层与维度（不可混用）

| 层级 | 适用 | 必填维度 | 可选维度 |
| --- | --- | --- | --- |
| **端点层** | 对外暴露的入口（被外部触发的处理单元） | `[接口地址]` `[功能描述]` `[业务逻辑]` | `[业务菜单]` `[前端路径]` `[涉及数据表]` `[涉及外部模块]` `[修改记录]` |
| **内部层** | 内部业务处理、工具、Helper、复杂私有方法 | `[核心目的]` `[业务逻辑]` | `[涉及数据表]` `[涉及外部模块]` `[修改记录]` |

**勿混用**：端点写 `[接口地址]`/`[功能描述]`；内部写 `[核心目的]`。同一段注释不得同时出现端点维与 `[核心目的]`。

## 维度定义（技术无关）

| 维度 | 含义 | 通用化说明 |
| --- | --- | --- |
| `[接口地址]` | **入口标识** | 任何唯一标识该入口的串：HTTP 路径、命令名、消息主题 / 事件名、对外公开的函数签名等，按实际形态填一种 |
| `[功能描述]` | 该入口对外完成的业务目标 | — |
| `[核心目的]` | 该内部方法解决的业务问题 | — |
| `[业务逻辑]` | 核心业务步骤 | 与方法体逐步对应；写「做什么 / 为什么」，**禁止泛化模板句** |
| `[涉及数据表]` | **涉及数据** | 任意持久化 / 状态资源：表、集合、文件、缓存键等，标注 `(R/W/D)`（读 / 写 / 删） |
| `[涉及外部模块]` | 依赖的外部服务、其它系统、组件 | 写服务 / 组件名 |
| `[业务菜单]` | 关联的功能菜单 | **仅当存在前端界面时填**，无界面则省略 |
| `[前端路径]` | 触发该入口的前端页面文件 | **仅当存在前端工程时填**，无则省略 |
| `[修改记录]` | 逻辑变更历史 | 逻辑变更时追加 |

## 核心原则

1. **语义优先**：`[业务逻辑]` 与方法体逐步对应，写「做什么 / 为什么」，**不复述代码**；不同方法不得复用同一段步骤。
2. **高层总结底层**：端点 / 高层的 `[业务逻辑]` 须**提炼**所调用内部方法的业务含义；**禁止**写「调用某某服务」「封装返回」这类空壳步骤。
3. **自下而上**：批量补注释**先写底层（工具 / 业务处理单元）再写高层（对外入口）**；底层未写清 `[核心目的]` + 逐步 `[业务逻辑]` 前，不写该链路高层终态。
4. **证据驱动**：入口标识、数据资源、菜单、前端路径**必须有证据**（查入口定义 / 前端调用 / 资源名）；无证据则**省略**该可选维，不写 `TODO: 待确认`，不写 `[涉及数据表]: 无`。
5. **只改注释不改逻辑**（批量任务同样）。
6. **保留**原有人工有效业务背景，只补缺失维度。

## 执行步骤

1. **判定方法类型**：端点 / 内部。
2. **取证**：定位入口标识、读方法体提炼业务步骤、确认数据资源 / 外部依赖；有前端界面时按下文「先文件后菜单」取证。
3. **生成 / 更新注释**：必填维齐全、表述精简，`[业务逻辑]` 与源码逐步对应。
4. **维护修改记录（仅逻辑变更时）**：`[修改记录]` 追加 `[YYYY-MM-DD] [需求名] - [简述]`，不删历史条目。
5. **自检**：注释与源码一致；无泛化模板句；无空壳调用句；可选维有证据才出现。

## [业务菜单] 与 [前端路径] 取证（先文件后菜单，仅有前端时）

**顺序固定：先定位前端文件，再反查菜单。**

```text
入口标识 → 查前端调用 → 定位前端页面文件 → [前端路径]
        → 该文件对应路由 → 菜单 → [业务菜单]
```

- 命中在接口封装模块时，继续查谁引用它，追到实际发起调用的页面文件。
- 找不到菜单时**只上溯一层**（子组件 → 父页面、公共方法 → 调用它的页面）再匹配；仍无则**省略 `[业务菜单]`**，不猜菜单名。
- 禁止：先看菜单配置去**猜**前端路径；`[业务菜单]` 与路由 path 不一致。

## 禁止的泛化 `[业务逻辑]`

以下短语不得作为最终注释（除非与实现完全一致且仅此一句）：

- `1. 解析入参 -> 2. 执行核心逻辑 -> 3. 返回结果`
- `1. 接收请求参数 -> 2. 执行业务处理 -> 3. 返回结果`
- `1. 组装查询条件 -> 2. 执行查询 -> 3. 返回结果集`（所有查询方法雷同时）
- 整句仅「调用某某服务」或「封装返回」

**格式禁令**：`[涉及数据表]: 无`、双冒号 `: :`。

## 格式示例（中性伪代码）

**端点层：**

```text
注释块 {
  [业务菜单]: 一级 -> 二级 -> 三级            # 可选，仅有前端界面时
  [前端路径]: <触发该入口的前端页面文件>       # 可选，仅有前端工程时
  [接口地址]: <入口标识：路径 / 命令 / 主题 / 函数签名>
  [功能描述]: <该入口对外完成的业务目标，一句话>
  [业务逻辑]: 1. <业务步骤一> -> 2. <业务步骤二> -> 3. <返回的业务结果>
  [涉及数据表]: <资源A> (R), <资源B> (W)。     # 可选，有证据时
  [涉及外部模块]: <外部服务 / 组件>。          # 可选，有证据时
}
```

**内部层：**

```text
注释块 {
  [核心目的]: <该方法解决的业务问题，一句话>
  [业务逻辑]: 1. <内部处理步骤一> -> 2. <步骤二> -> 3. <产出>
  [涉及数据表]: <资源> (R/W/D)。              # 可选
  [涉及外部模块]: <外部服务 / 组件>。          # 可选
}
```

> 注释块用各语言的标准注释语法承载（行注释 / 块注释 / 文档注释皆可）；维度顺序与标签保持一致，逐行书写。

## 强制约束

- **严禁臆测**：入口标识、数据资源、菜单必须有证据。
- **可选维无证据则省略**：不得 `TODO: 待确认`，不得 `[涉及数据表]: 无`。
- **语义优先于结构**：维度齐全 ≠ 合格，`[业务逻辑]` 必须是真实业务语义。
- **精简优先**：只写规范要求的维度。
- **存量补全**：改代码时顺带补全该方法注释。
- **批量自下而上**：底层注释先于高层；高层 `[业务逻辑]` 须引用底层注释，非空壳调用句。

## 完成检查

- [ ] 方法类型判定正确；端点 / 内部维度未混用
- [ ] `[业务逻辑]` 与源码逐步对应；无「仅调用某某服务」空壳句
- [ ] 端点 `[业务逻辑]` 为所调用内部方法的业务摘要（高层总结底层）
- [ ] 批量任务：底层（工具 / 业务处理单元）已先于高层（对外入口）完成
- [ ] 有前端时：`[前端路径]` 由入口标识查到页面文件；`[业务菜单]` 由该文件路由匹配（仍无则省略）
- [ ] 无泛化模板句、无 `[涉及数据表]: 无`
- [ ] 逻辑有变更时已追加 `[修改记录]`
- [ ] 仅改注释，未改业务逻辑
